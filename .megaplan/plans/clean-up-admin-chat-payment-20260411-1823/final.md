## Meta

Land in dependency order: T1 and T2 are independent and unblock everything; T3 is also independent and small. T4 and T5 MUST land together \u2014 T4 reuses receipt_prompt_message_id to point at the status message, and T5 is the only thing that stops _cleanup_admin_intent_messages from deleting it on admin approval. If you land T4 without T5 you WILL delete the shared status message mid-flow. T6 should be updated alongside T4/T5 and should not be merged without green tests.

The nested-try retry in T2 is the subtlety that makes FLAG-002 safe: the outer try/except \u2192 return None contract stays identical, so every caller that branches on a falsey return (e.g. admin_chat_cog.py:703-710, :771-773) keeps its fail-closed recovery path. Do not refactor this into a helper \u2014 inline is the point.

The _set_intent_status_message helper should stay ~40 lines. If it's growing past that, you are probably duplicating send logic or over-abstracting; simplify. It is the SINGLE point of creation/edit for the status message within AdminChatCog.

For the final reply, use message.reply(...) (not channel.send with a reference) so the Discord client renders the reply affordance. mention_author=True ensures the tag actually pings. suppress_embeds=True is the key piece that stops the Solana Explorer link from rendering a card \u2014 without it the whole collapse is defeated.

Status wording: default to concise single-line transitions \u2014 'Verifying wallet with a small test payment\u2026' \u2192 '<@recipient> Test payment confirmed. Please reply here with `confirmed` once you see it in your wallet.' \u2192 'Confirmation received. Awaiting admin approval for the final payout.' \u2192 'Payout queued for sending.' (from payment_ui_cog on admin approval) \u2192 'Payout sending\u2026' \u2192 reply 'Payment sent \u2014 X.XXXX SOL. <explorer_link>'. The plan authors flagged these are replaceable \u2014 if the user asks for different wording during execution, change the strings but not the structure.

Failure branch default: edit the status message to a short terminal failure line rather than going silent. Assumption #5 in the plan metadata. If the user redirects, only the string changes.

Be careful with the test-confirmed transition's reuse of receipt_prompt_message_id. This works only because T5's cleanup exclusion filters status_message_id out of the delete list. If you reorder implementation and ship T4 alone, the cleanup sweep WILL delete the shared status message on admin approval.

Do NOT widen worker-side suppression to other producers. The explicit success criterion requires non-admin_chat producers to keep receiving the generic result card. The negative test (test_handle_payment_result_still_notifies_non_admin_chat_producers) exists to guard this.

For T7, the throwaway repro script lives under /tmp and is deleted after it runs. Do not commit it, do not add it to tests/. It exists purely to give the executor ground-truth confirmation that the channel collapse works end-to-end at runtime (beyond what unit tests assert).
